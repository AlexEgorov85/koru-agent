# DOCUMENTATION_POLICY.md

## Соглашения о документации проекта Agent_v5

### 1. Управление устаревшими документами
- Промежуточные отчеты удаляются после финализации
- Все архитектурные решения фиксируются в `ARCHITECTURE_DECISIONS.md`
- Отчеты о соответствии архивируются с датой и версией
- Старые структуры данных перемещаются в `archive/` перед удалением

### 2. Структура документации
- Основная архитектурная документация находится в корне проекта
- Руководства пользователя и разработчика находятся в `docs/guides/`
- Техническая документация по компонентам в `docs/architecture/`
- Финальные отчеты и аналитика в `reports/`

### 3. Формат и стиль документации
- Все документы используют формат Markdown
- Для диаграмм использовать Mermaid
- Для таблиц использовать формат Markdown
- Все даты указываются в формате DD MMMM YYYY (например, 15 февраля 2026)

### 4. Управление версиями документации
- Каждый документ должен содержать дату последнего обновления
- При обновлении документа обновляется дата в конце файла
- История изменений ведется в `CHANGELOG.md`
- Критические изменения документируют в `ARCHITECTURE_DECISIONS.md`

### 5. Проверка актуальности
- Архитектурные документы проверяются каждые 2 недели
- Статус актуальности отражается в `ARCHITECTURE_CHECKLIST.md`
- Устаревшие документы помечаются как "OBSOLETE" с датой
- Устаревшие документы удаляются через 1 месяц после пометки

### 6. Удаление устаревших файлов
- Перед удалением убедиться, что вся важная информация перенесена
- Удаление производится только после согласования с командой
- Удаленные файлы должны быть исключены из истории Git при необходимости
- Удаление `_old_*` директорий производится после проверки миграции

### 7. Обновление документации
- При каждом архитектурном изменении обновляется соответствующая документация
- Обновления документации сопровождаются комментариями о причинах изменений
- При рефакторинге компонентов обновляется `ARCHITECTURE.md`
- При изменении интерфейсов обновляются соответствующие руководства

### 8. Ответственность
- Владелец компонента отвечает за актуальность его документации
- Архитектор отвечает за общую согласованность документации
- Каждый разработчик может предлагать улучшения документации
- Обновления документации рассматриваются на архитектурных ревью
