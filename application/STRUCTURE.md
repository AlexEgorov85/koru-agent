# Структура слоя приложения

## Принципы организации
Слой приложения содержит ТОЛЬКО компоненты, зависящие от абстракций домена. Никаких прямых зависимостей от инфраструктуры (файловая система, сеть, конкретные реализации).

### Категории компонентов

1. **`thinking_patterns/`** — Стратегии решения задач (паттерны мышления)
   - `base/` — универсальные паттерны (ReAct, Planning, Reflection)
   - `specialized/` — доменно-специфичные паттерны (анализ безопасности, кода)
   - `composable/` — компонуемые паттерны из базовых строительных блоков
   - Зависят ТОЛЬКО от `domain.abstractions.*` (ITool, ISkill, IThinkingPattern)
   - НЕ зависят от инфраструктуры (ФС, сеть, конкретные реализации)

2. **`orchestration/`** — Высокоуровневая координация
   - `agent_orchestrator.py` — координация выполнения агента через абстракции
   - `agent_service.py` — сервис оркестрации задач
   - `pattern_executor.py` — выполнение паттернов через абстракции
   - Зависит ТОЛЬКО от `domain.abstractions.*` и `thinking_patterns/`

3. **`coordination/`** — Управление состоянием и восстановлением
   - `pattern_recovery_manager.py` — восстановление после ошибок
   - `pattern_state_manager.py` — управление состоянием паттернов
   - Содержит ТОЛЬКО логику координации, без инфраструктурных деталей

4. **`services/`** — Доменные сервисы (чистая бизнес-логика)
   - `solution_analyzer.py` — анализ путей решения (без ФС/сети)
   - `prompt_renderer.py` — рендеринг через абстракции репозитория
   - НЕ содержат прямой работы с файловой системой или сетью

5. **`context/`** — Контексты (ТОЛЬКО данные)
   - `session_context.py` — состояние сессии (чистые данные)
   - `system_context.py` — системный контекст (очищен от реестров)
   - НЕ содержат зависимостей от инфраструктуры

6. **`bootstrap/`** — Инициализация и создание зависимостей (граница)
   - `application_bootstrap.py` — инициализация системы
   - `factories/` — фабрики создания компонентов
   - ЕДИНСТВЕННОЕ место, где допустимо создание инфраструктурных компонентов
   - Должно быть изолировано от основной логики приложения

## Маркерная папка `to_infrastructure/`
Содержит компоненты, которые НЕ должны находиться в слое приложения:
- Загрузчики промтов/паттернов — работа с ФС → `infrastructure/services/prompt_storage/`
- Реестры инструментов/навыков — реализации портов → `infrastructure/registries/`
- Статические промты — шаблоны → `infrastructure/prompts/templates/`
- Runtime с инфраструктурными зависимостями → `infrastructure/orchestration/`

## Маркерная папка `to_domain/`
Содержит компоненты, которые НЕ должны находиться в слое приложения - модели данных, абстрактные классы, интерфейсы

## Запрещено в слое приложения
- Прямые импорты из `infrastructure.*`
- Работа с файловой системой (`open()`, `Path`, `os`)
- Работа с сетью (HTTP-клиенты, сокеты)
- Конкретные реализации портов (только абстракции)
- Хранение статических данных (промты, конфиги) — только ссылки на репозитории

## Почему паттерны мышления в приложении, а не в домене или инфраструктуре?

| Слой | Почему НЕ подходит |
|------|-------------------|
| **`domain/`** | Паттерны — это **поведение/стратегия**, а не чистые данные. Домен содержит сущности и правила, но не стратегии оркестрации. |
| **`infrastructure/`** | Паттерны **не являются адаптерами** к внешним системам. Они не зависят от ФС/сети/БД — используют только абстракции (`ITool`, `ISkill`). |
| **`application/`** | ✅ Паттерны — это **координационная логика**, которая использует абстракции домена для решения задач. Это именно то, что должно быть в слое приложения. |

    def __init__(self, tool: ITool, skill: ISkill):
