# Архитектурное улучшение: LLMOrchestrator для управления вызовами LLM

## Проблема

В исходной реализации существовал серьёзный архитектурный пробел:

1. **Потеря ответа при таймауте**: При превышении таймаута LLM-вызова агент получал исключение `TimeoutError`, которое пробрасывалось вверх по стеку вызовов.

2. **"Висячие" фоновые потоки**: Синхронный вызов LLM выполнялся в потоке через `run_in_executor`. При таймауте:
   - Исключение обрабатывалось на уровне агента
   - Фоновый поток **продолжал выполнение** и потреблял ресурсы
   - Результат вызова **терялся** (никто его не использовал)
   - Пул потоков мог переполниться "брошенными" задачами

3. **Отсутствие мониторинга**: Не было возможности отследить:
   - Сколько вызовов выполняется в данный момент
   - Сколько вызовов превысило таймаут
   - Сколько "брошенных" вызовов завершилось позже
   - Среднее время выполнения вызовов

4. **Некорректная обработка ошибок**: Исключения от LLM приводили к падению агента или непредсказуемому поведению.

## Решение: LLMOrchestrator

Создан централизованный оркестратор для управления всеми LLM вызовами.

### Ключевые компоненты

#### 1. CallStatus - Статусы вызова
```python
class CallStatus(str, Enum):
    PENDING = "pending"           # Ожидает запуска
    RUNNING = "running"           # Выполняется
    COMPLETED = "completed"       # Успешно завершён
    TIMED_OUT = "timed_out"       # Превышен таймаут (но поток ещё работает)
    FAILED = "failed"             # Ошибка выполнения
    CANCELLED = "cancelled"       # Отменён пользователем
```

#### 2. CallRecord - Запись о вызове
Хранит полную информацию о каждом вызове:
- Уникальный ID
- Статус выполнения
- Время начала/окончания
- Результат или ошибка
- Имя потока

#### 3. LLMMetrics - Метрики
Отслеживает:
- Общее количество вызовов
- Успешные/неудачные вызовы
- Таймауты
- "Брошенные" вызовы (orphaned)
- Среднее время генерации
- Процент таймаутов

### Архитектура работы

```
┌─────────────────────────────────────────────────────────────┐
│                    ReActPattern                              │
│  generate_decision()                                         │
│    └─ _perform_structured_reasoning()                        │
│         └─ _execute_llm_with_orchestrator()                  │
└────────────────────────┬────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────────┐
│                   LLMOrchestrator                            │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  execute(request, timeout, provider)                 │   │
│  │    1. Генерация call_id                              │   │
│  │    2. Регистрация в реестре                          │   │
│  │    3. Запуск в executor                              │   │
│  │    4. Ожидание с таймаутом                           │   │
│  │    5. Возврат LLMResponse (с error при таймауте)     │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  Реестр активных вызовов                             │   │
│  │  { call_id: CallRecord }                             │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  Фоновая очистка (cleanup_loop)                      │   │
│  │  - Удаляет старые записи                             │   │
│  │  - Логирует "брошенные" вызовы                       │   │
│  └──────────────────────────────────────────────────────┘   │
└────────────────────────┬────────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────────┐
│              ThreadPoolExecutor (4 worker)                   │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐        │
│  │ Поток 1 │  │ Поток 2 │  │ Поток 3 │  │ Поток 4 │        │
│  │ LLM     │  │ LLM     │  │ LLM     │  │ LLM     │        │
│  │ вызов   │  │ вызов   │  │ вызов   │  │ вызов   │        │
│  └─────────┘  └─────────┘  └─────────┘  └─────────┘        │
└─────────────────────────────────────────────────────────────┘
```

### Обработка таймаута

```
1. Запуск вызова
   │
   ▼
2. Ожидание с таймаутом (asyncio.wait_for)
   │
   ├─► Успех ──────► Возврат LLMResponse
   │
   └─► Таймаут
        │
        ▼
   3. Пометка вызова как TIMED_OUT
        │
        ▼
   4. Возврат LLMResponse с metadata.error
        │
        ▼
   5. Фоновый поток завершается
        │
        ├─► Если статус TIMED_OUT ──► Логирование orphaned call
        │
        └─► Если статус RUNNING ────► Обновление на COMPLETED
```

### Интеграция в ReActPattern

```python
# В _perform_structured_reasoning():

orchestrator = self.llm_orchestrator

if orchestrator:
    # Вызов через оркестратор
    success, resp, error = await self._execute_llm_with_orchestrator(
        llm_request=llm_request,
        llm_provider=llm_provider,
        timeout=llm_timeout,
        session_context=session_context
    )
    
    if success:
        response = resp
    else:
        # Обработка ошибки без падения агента
        return fallback_decision
else:
    # Fallback: прямой вызов (для обратной совместимости)
    response = await asyncio.wait_for(
        llm_provider.generate_structured(llm_request),
        timeout=llm_timeout
    )
```

### Обработка ошибок на уровне агента

```python
# В AgentRuntime._execute_single_step_internal():

execution_result = await self.executor.execute_capability(...)

# Проверка на ошибку
if hasattr(execution_result, 'metadata') and execution_result.metadata:
    error_msg = execution_result.metadata.get('error')
    if error_msg:
        # Переключение на fallback паттерн
        return BehaviorDecision.switch_to_fallback(reason=error_msg)
```

## Преимущества

### 1. Управление ресурсами
- ✅ Ограниченное количество потоков (max_workers=4)
- ✅ Лимит ожидающих вызовов (max_pending_calls=100)
- ✅ Периодическая очистка старых записей

### 2. Мониторинг
- ✅ Метрики в реальном времени
- ✅ Статус здоровья оркестратора
- ✅ Логирование "брошенных" вызовов

### 3. Надёжность
- ✅ Агент не падает при таймаутах LLM
- ✅ Переключение на fallback паттерны
- ✅ Graceful degradation

### 4. Отладка
- ✅ Уникальный ID для каждого вызова
- ✅ Трассировка через event bus
- ✅ Детальные метрики для анализа

## Метрики для мониторинга

```python
metrics = orchestrator.get_metrics()
print(f"Всего вызовов: {metrics.total_calls}")
print(f"Успешных: {metrics.completed_calls}")

print(f"Среднее время: {metrics.avg_generation_time:.2f}с")

```

## Настройка

### В ApplicationContext
```python
self.llm_orchestrator = LLMOrchestrator(
    event_bus=self.infrastructure_context.event_bus,
    max_workers=4,           # Количество потоков
    cleanup_interval=60.0,   # Интервал очистки (сек)
    max_pending_calls=100    # Лимит очереди
)
```

### Таймауты
- По умолчанию: 120 секунд
- Настраивается в `LLMRequest.metadata['timeout']`
- Или в конфигурации провайдера `timeout_seconds`

## Обратная совместимость

LLMOrchestrator **не ломает** существующий код:
- Если оркестратор недоступен, используется прямой вызов LLM
- ReActPattern автоматически переключается на fallback
- Все существующие тесты продолжают работать

## Заключение

LLMOrchestrator устраняет критический архитектурный пробел в управлении LLM вызовами:
- **Предотвращает потерю ресурсов** через контроль потоков
- **Обеспечивает надёжность** через обработку ошибок без исключений
- **Даёт наблюдаемость** через метрики и мониторинг
- **Сохраняет гибкость** через обратную совместимость

Это точечное, но важное улучшение, которое делает систему устойчивой к таймаутам и перегрузкам.
