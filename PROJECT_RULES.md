# Руководящие принципы разработки для проекта Koru AI Agent Framework

## 🛡️ Правила безопасности

1. **Запрет разрушительных операций без подтверждения**
   - Не удалять/переименовывать корневые директории
   - Не изменять конфигурацию сборки и контроля версий (`.git`, `package.json`, `requirements.txt`, `pyproject.toml`, `Dockerfile`) без явного одобрения

2. **Не изменять публичные контракты**
   - Сигнатуры API, интерфейсы, форматы данных и базовые абстракции — только после согласования
   - Внутреннюю реализацию можно исправлять свободно

3. **Запрет побочных эффектов**
   - Никаких `eval()`, динамических импортов из пользовательского ввода
   - Запись только в пределах рабочей директории проекта

## 🧪 Тестирование

4. **Проверять изменения через тесты**
   - Если в проекте есть тесты — после любых изменений запускать релевантные (модульные для изменённого файла, интеграционные для кросс-модульных правок)
   - При падении — откатывать и сообщать пользователю

5. **Добавлять тесты для нового кода**
   - Исправление бага → добавить регрессионный тест
   - Новый функционал → покрыть тестами в стиле существующих (расположение, именование, фреймворк)

6. **Не отключать и не игнорировать тесты**
   - Удаление/пропуск тестов — только по прямому запросу пользователя с указанием причины

## 📝 Стиль кода и комментарии

7. **Анализировать стиль ДО изменений**
   - Перед правкой изучить 2–3 соседних файла:
     - стиль импортов и именования
     - **формат комментариев** (язык, регистр, пунктуация, отступы)
     - организацию логики и слоёв
   - Новый код и комментарии должны быть неотличимы от существующих

8. **Правила для комментариев на русском**
   - **Единый язык**: если проект использует русские комментарии — все новые комментарии только на русском. Не смешивать языки в рамках файла
   - **Формат**: предложения с заглавной буквы, точка в конце. Для однострочных — без лишних отступов (`# Объясняем почему`, а не `#  Объясняем почему  `)
   - **Когда комментировать**: только нетривиальную логику, «магические» значения, временные решения. Не дублировать очевидное (`# увеличиваем счётчик` над `i += 1`)
   - **Структурированные блоки**: для сложных алгоритмов — блочный комментарий с кратким описанием цели и ключевых шагов
   - **TODO/FIXME**: только с указанием причины и (если известно) сроком/условием завершения:
     ```python
     # TODO(@алексей): заменить хардкод после миграции конфига (до релиза 2.0)
     ```

## 🧭 Архитектурная дисциплина

9. **Соблюдать границы слоёв**
   - Не перемещать логику между слоями (бизнес-логика ↔ утилиты ↔ презентация)
   - Новый код размещать строго в соответствии с архитектурой проекта

10. **Не вводить новые архитектурные паттерны**
    - Если проект не использует фабрики/репозитории/DDD — не добавлять их «для красоты»
    - Расширять существующие абстракции, а не создавать новые

## 🔁 Процесс изменений

11. **Итеративный подход**
    - Нетривиальные изменения предлагать в виде диффа → ждать подтверждения → применять атомарно

12. **Минимальные правки**
    - Исправлять только запрошенное. Не рефакторить «на всякий случай» соседний код

13. **Верификация синтаксиса**
    - После изменений — проверить через парсер/AST: нет ли ошибок синтаксиса, битых импортов, незакрытых блоков

14. **Разрешено без подтверждения**
    - Исправление багов в реализации
    - Добавление приватных утилит/хелперов
    - Создание файлов по аналогии с соседними
    - Типизация, документация, тесты, комментарии в стиле проекта

15. **При сомнении — уточнить**
    - Если структура или стиль неочевидны — остановиться и задать вопрос пользователю

## 🏗️ СТРАТЕГИЯ РАЗДЕЛЕНИЯ КОМПОНЕНТОВ

16. **ИНСТРУМЕНТЫ** (infrastructure/tools/) — низкоуровневые операции
    - Только чтение/запись файлов
    - Только парсинг в сыром виде
    - Никакой бизнес-логики

17. **СЕРВИСЫ** (infrastructure/services/) — обработка данных
    - Анализ AST для извлечения структуры
    - Разрешение зависимостей
    - Построение графов
    - Никакого взаимодействия с пользователем

18. **НАВЫКИ** (infrastructure/adapters/skills/) — оркестрация
    - Комбинирование инструментов и сервисов
    - Формирование ответов для агента
    - Управление контекстом выполнения

## 📋 ПРИНЦИПЫ SOLID

19. **S — Единственная ответственность (Single Responsibility)**
    - Каждый класс/модуль должен решать только одну задачу
    - Если класс содержит логику работы с файловой системой + парсинг данных + валидация → РАЗДЕЛИТЬ на 3 класса
    - Комментарий-маркер для рефакторинга:
      ```python
      # TODO: [SOLID:S] Класс нарушает SRP — содержит бизнес-логику и работу с БД. Вынести в отдельный репозиторий.
      ```

20. **O — Открытость/Закрытость (Open/Closed)**
    - Классы должны быть открыты для расширения, закрыты для модификации
    - Предпочитать наследование через абстрактные базовые классы или интерфейсы
    - Пример комментария:
      ```python
      # Расширяем функционал через наследование, не изменяя базовый класс
      # См. принцип открытости/закрытости (SOLID:O)
      ```

21. **L — Подстановка Лисков (Liskov Substitution)**
    - Объекты базового типа должны заменяться объектами производных типов без нарушения логики
    - Запрещено:
      * переопределять методы с изменением контракта
      * выбрасывать исключения в производных классах, которых нет в базовом
    - Комментарий для проверки:
      ```python
      # Проверка: производный класс должен корректно заменять базовый (SOLID:L)
      ```

22. **I — Разделение интерфейсов (Interface Segregation)**
    - Не создавать «жирные» интерфейсы с методами, не нужными конкретному клиенту
    - Разделять интерфейсы по группам ответственности:
      * `IReadable` / `IWritable` вместо `IFileHandler`
    - Комментарий:
      ```python
      # Интерфейс разделён по зонам ответственности (SOLID:I)
      # Клиенты зависят только от нужных им методов
      ```

23. **D — Инверсия зависимостей (Dependency Inversion)**
    - Модули верхнего уровня не должны зависеть от модулей нижнего уровня
    - Оба должны зависеть от абстракций
    - Абстракции не должны зависеть от деталей. Детали должны зависеть от абстракций
    - Использовать внедрение зависимостей (DI):
      ```python
      # Зависимость инвертирована через интерфейс (SOLID:D)
      # Реализация передаётся извне, а не создаётся внутри класса
      ```

## 📝 Требования к комментариям на русском

24. **Формат комментариев**
    - **Класс**: `"""Краткое описание назначения класса"""`  
    - **Публичный метод**: `"""Что делает, зачем нужен, особенности поведения"""`  
    - **Сложная логика**: `# Пояснение алгоритма или нетривиального решения`  
    - **Нарушение SOLID**: `# FIXME: [SOLID:X] Описание проблемы и решение`  
    - **Рефакторинг**: `# TODO: [SOLID:X] План улучшения`  

## ⛔ Запрещённые паттерны

25. **Запрещённые анти-паттерны**:
    - ❌ Бог-объекты (классы с 10+ публичными методами разной ответственности)
    - ❌ Жёсткая привязка к конкретным реализациям (new ConcreteClass() внутри бизнес-логики)
    - ❌ Интерфейсы с методами, которые клиенты не используют
    - ❌ Изменение поведения базового класса в наследниках без сохранения контракта

## 🏗️ Архитектурные принципы проекта

26. **Чистая архитектура (Clean Architecture)**
    - Доменный слой (domain/) содержит бизнес-логику и правила
    - Прикладной слой (application/) координирует работу компонентов домена
    - Инфраструктурный слой (infrastructure/) реализует внешние зависимости
    - Зависимости направлены внутрь: внешние слои зависят от внутренних, а не наоборот

27. **Инверсия зависимостей**
    - Приложение зависит от абстракций, а не от конкретных реализаций
    - Конкретные реализации инжектируются в приложение через интерфейсы
    - Это позволяет легко заменять реализации без изменения бизнес-логики

28. **Компонуемость**
    - Возможность комбинировать различные паттерны мышления для решения сложных задач
    - Паттерны мышления должны быть независимыми и переиспользуемыми
    - Атомарные действия должны быть минимальными неделимыми операциями
