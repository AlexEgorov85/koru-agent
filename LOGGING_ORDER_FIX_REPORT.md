# Отчёт о реализации исправления порядка логирования

## Дата: 9 марта 2026 г.

## Проблема
Асинхронная публикация событий логирования через EventBus приводила к нарушению порядка вывода логов во время инициализации компонентов. Логи могли выводиться в произвольном порядке, что затрудняло отладку.

## Решение
Реализована комбинация **Решения 1** (синхронные логи в инициализации) и **Решения 4** (переключение режима после инициализации) с улучшениями:

### Ключевые изменения

#### 1. EventBusLogger (`core/infrastructure/logging/logger.py`)

**Добавлено:**
- `LoggerInitializationState` — enum состояний инициализации
- `_init_state` — отслеживание текущего состояния
- `_get_init_state_callback` — callback для автоматического определения состояния компонента
- `_is_initializing()` — проверка фазы инициализации
- `_set_initializing()` / `_set_ready()` — переключение режимов
- `_write_sync()` — синхронная запись в stdout/stderr с обработкой ошибок

**Изменено:**
- Методы `info()`, `debug()`, `warning()`, `error()`, `exception()` теперь автоматически выбирают режим:
  - **INITIALIZING/NOT_INITIALIZED** → синхронный вывод в stdout/stderr
  - **READY** → асинхронная публикация через EventBus
- Методы `*_sync()` теперь означают "принудительно синхронный вывод", а не "синхронная публикация в EventBus"

**Обработка ошибок:**
- Если запись в stdout не удалась, происходит fallback на публикацию через EventBus
- Это предотвращает потерю логов при проблемах с терминалом

#### 2. BaseComponent (`core/components/base_component.py`)

**Добавлено:**
- `_get_logger_init_state()` — callback для получения состояния компонента
- Интеграция с EventBusLogger через callback

**Изменено:**
- `_init_event_bus_logger()` — передаёт callback в конструктор логгера
- `initialize()` — явно переключает логгер в режим инициализации, затем в READY

**Жизненный цикл:**
```
CREATED → INITIALIZING (синхронный вывод) → READY (асинхронный) → SHUTDOWN
```

#### 3. Тесты (`tests/unit/infrastructure/test_event_bus_logger.py`)

**Добавлено:**
- `TestEventBusLoggerSync` — тесты синхронного вывода в stdout/stderr
- `TestEventBusLoggerAsync::test_auto_mode_initializing` — тест автоматического выбора режима
- `TestEventBusLoggerAsync::test_auto_mode_ready` — тест асинхронной публикации
- `TestEventBusLoggerAutoModeSwitching` — тесты переключения режимов

**Все 14 тестов проходят.**

## Архитектура

```
┌─────────────────────────────────────────────────────────────┐
│                    Компонент (BaseComponent)                │
│  ┌───────────────────────────────────────────────────────┐  │
│  │  Lifecycle: CREATED → INITIALIZING → READY            │  │
│  │       ↓ callback                                       │  │
│  │  EventBusLogger._get_logger_init_state()              │  │
│  └───────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│                    EventBusLogger                           │
│  ┌───────────────────────────────────────────────────────┐  │
│  │  _is_initializing()?                                   │  │
│  │       ↓ YES                    ↓ NO                    │  │
│  │  _write_sync()            _publish()                   │  │
│  │  (stdout/stderr)          (EventBus)                   │  │
│  │       ↓                        ↓                       │  │
│  │  [timestamp] [level]      Event → Queue → Worker      │  │
│  │  [component] message      → Subscriber                │  │
│  │  └─ fallback → EventBus   (TerminalHandler, etc.)     │  │
│  └───────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

## Преимущества

1. **Гарантия порядка логов** — во время инициализации логи выводятся строго по порядку
2. **Автоматическое переключение** — не требует явных вызовов от разработчика
3. **Производительность** — после инициализации используется асинхронная публикация
4. **Надёжность** — fallback предотвращает потерю логов
5. **Обратная совместимость** — существующий код продолжает работать

## Риски и ограничения

1. **Блокировка event loop** — синхронная запись может замедлить инициализацию при большом количестве логов
   - **Митигация:** инициализация происходит до обработки запросов, это допустимо
2. **Упрощённый формат** — синхронный вывод не включает extra данные
   - **Митигация:** extra данные не критичны для отладки инициализации
3. **Состояние в логгере** — требует аккуратного управления
   - **Митигация:** состояние синхронизировано с состоянием компонента через callback
