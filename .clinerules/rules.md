### 🛡️ Правила безопасности
1. **Запрет разрушительных операций без подтверждения**  
   Не удалять/переименовывать корневые директории, не изменять конфигурацию сборки и контроля версий (`.git`, `package.json`, `requirements.txt`, `pyproject.toml`, `Dockerfile`) без явного одобрения.

2. **Не изменять публичные контракты**  
   Сигнатуры API, интерфейсы, форматы данных и базовые абстракции — только после согласования. Внутреннюю реализацию можно исправлять свободно.

3. **Запрет побочных эффектов**  
   Никаких `eval()`, динамических импортов из пользовательского ввода, записи вне рабочей директории проекта.

---

### 🧪 Тестирование
4. **Проверять изменения через тесты**  
   Если в проекте есть тесты — после любых изменений запускать релевантные (модульные для изменённого файла, интеграционные для кросс-модульных правок). При падении — откатывать и сообщать пользователю.

5. **Добавлять тесты для нового кода**  
   Исправление бага → добавить регрессионный тест. Новый функционал → покрыть тестами в стиле существующих (расположение, именование, фреймворк).

6. **Не отключать и не игнорировать тесты**  
   Удаление/пропуск тестов — только по прямому запросу пользователя с указанием причины.

---

### 📝 Стиль кода и комментарии (объединённый блок)
7. **Анализировать стиль ДО изменений**  
   Перед правкой изучить 2–3 соседних файла:  
   — стиль импортов и именования  
   — **формат комментариев** (язык, регистр, пунктуация, отступы)  
   — организацию логики и слоёв  
   Новый код и комментарии должны быть неотличимы от существующих.

8. **Правила для комментариев на русском**  
   — **Единый язык**: если проект использует русские комментарии — все новые комментарии только на русском. Не смешивать языки в рамках файла.  
   — **Формат**: предложения с заглавной буквы, точка в конце. Для однострочных — без лишних отступов (`# Объясняем почему`, а не `#  Объясняем почему  `).  
   — **Когда комментировать**: только нетривиальную логику, «магические» значения, временные решения. Не дублировать очевидное (`# увеличиваем счётчик` над `i += 1`).  
   — **Структурированные блоки**: для сложных алгоритмов — блочный комментарий с кратким описанием цели и ключевых шагов.  
   — **TODO/FIXME**: только с указанием причины и (если известно) сроком/условием завершения:  
     ```python
     # TODO(@алексей): заменить хардкод после миграции конфига (до релиза 2.0)
     ```

### 🧭 Архитектурная дисциплина
9. **Соблюдать границы слоёв**  
   Не перемещать логику между слоями (бизнес-логика ↔ утилиты ↔ презентация). Новый код размещать строго в соответствии с архитектурой проекта.

10. **Не вводить новые архитектурные паттерны**  
    Если проект не использует фабрики/репозитории/DDD — не добавлять их «для красоты». Расширять существующие абстракции, а не создавать новые.

---

### 🔁 Процесс изменений
11. **Итеративный подход**  
    Нетривиальные изменения предлагать в виде диффа → ждать подтверждения → применять атомарно.

12. **Минимальные правки**  
    Исправлять только запрошенное. Не рефакторить «на всякий случай» соседний код.

13. **Верификация синтаксиса**  
    После изменений — проверить через парсер/AST: нет ли ошибок синтаксиса, битых импортов, незакрытых блоков.

14. **Разрешено без подтверждения**  
    — Исправление багов в реализации  
    — Добавление приватных утилит/хелперов  
    — Создание файлов по аналогии с соседними  
    — Типизация, документация, тесты, комментарии в стиле проекта

15. **При сомнении — уточнить**  
    Если структура или стиль неочевидны — остановиться и задать вопрос пользователю.

16. **СТРАТЕГИЯ РАЗДЕЛЕНИЯ**:

 1. ИНСТРУМЕНТЫ (infrastructure/tools/) — низкоуровневые операции
    - Только чтение/запись файлов
    - Только парсинг в сыром виде
    - Никакой бизнес-логики

 2. СЕРВИСЫ (infrastructure/services/) — обработка данных
    - Анализ AST для извлечения структуры
    - Разрешение зависимостей
    - Построение графов
    - Никакого взаимодействия с пользователем

 3. НАВЫКИ (infrastructure/adapters/skills/) — оркестрация
    - Комбинирование инструментов и сервисов
    - Формирование ответов для агента
    - Управление контекстом выполнения
