# ПРАВИЛА РАБОТЫ С ПРОЕКТОМ (SOLID + ЛОКАЛИЗАЦИЯ)

## 🔷 ОБЩИЕ ТРЕБОВАНИЯ
1. Все публичные методы, классы и интерфейсы должны содержать документирующие комментарии на русском языке.
2. Комментарии должны объяснять «почему», а не «что» делает код.
3. Избегать избыточных комментариев для тривиального кода.
4. Для сложной логики добавлять блочные комментарии с пояснением алгоритма.

## 🔷 ПРИНЦИП S — ЕДИНСТВЕННАЯ ОТВЕТСТВЕННОСТЬ
- Каждый класс/модуль должен решать только одну задачу.
- Если класс содержит логику:
  * работы с файловой системой + парсинг данных + валидация → РАЗДЕЛИТЬ на 3 класса
- Комментарий-маркер для рефакторинга:
  ```python
  # TODO: [SOLID:S] Класс нарушает SRP — содержит бизнес-логику и работу с БД. Вынести в отдельный репозиторий.
  ```

## 🔷 ПРИНЦИП O — ОТКРЫТОСТЬ/ЗАКРЫТОСТЬ
- Классы должны быть открыты для расширения, закрыты для модификации.
- Предпочитать наследование через абстрактные базовые классы или интерфейсы.
- Пример комментария:
  ```python
  # Расширяем функционал через наследование, не изменяя базовый класс
  # См. принцип открытости/закрытости (SOLID:O)
  ```

## 🔷 ПРИНЦИП L — ПОДСТАНОВКА ЛИСКОВ
- Объекты базового типа должны заменяться объектами производных типов без нарушения логики.
- Запрещено:
  * переопределять методы с изменением контракта
  * выбрасывать исключения в производных классах, которых нет в базовом
- Комментарий для проверки:
  ```python
  # Проверка: производный класс должен корректно заменять базовый (SOLID:L)
  ```

## 🔷 ПРИНЦИП I — РАЗДЕЛЕНИЕ ИНТЕРФЕЙСОВ
- Не создавать «жирные» интерфейсы с методами, не нужными конкретному клиенту.
- Разделять интерфейсы по группам ответственности:
  * `IReadable` / `IWritable` вместо `IFileHandler`
- Комментарий:
  ```python
  # Интерфейс разделён по зонам ответственности (SOLID:I)
  # Клиенты зависят только от нужных им методов
  ```

## 🔷 ПРИНЦИП D — ИНВЕРСИЯ ЗАВИСИМОСТЕЙ
- Модули верхнего уровня не должны зависеть от модулей нижнего уровня.
- Оба должны зависеть от абстракций.
- Абстракции не должны зависеть от деталей. Детали должны зависеть от абстракций.
- Использовать внедрение зависимостей (DI):
  ```python
  # Зависимость инвертирована через интерфейс (SOLID:D)
  # Реализация передаётся извне, а не создаётся внутри класса
  ```

## 🔷 ТРЕБОВАНИЯ К КОММЕНТАРИЯМ НА РУССКОМ
| Элемент          | Формат комментария                                  |
|------------------|-----------------------------------------------------|
| Класс            | `"""Краткое описание назначения класса"""`          |
| Публичный метод  | `"""Что делает, зачем нужен, особенности поведения"""` |
| Сложная логика   | `# Пояснение алгоритма или нетривиального решения`  |
| Нарушение SOLID  | `# FIXME: [SOLID:X] Описание проблемы и решение`    |
| Рефакторинг      | `# TODO: [SOLID:X] План улучшения`                  |

## 🔷 ЗАПРЕЩЁННЫЕ ПАТТЕРНЫ
❌ Бог-объекты (классы с 10+ публичными методами разной ответственности)
❌ Жёсткая привязка к конкретным реализациям (new ConcreteClass() внутри бизнес-логики)
❌ Интерфейсы с методами, которые клиенты не используют
❌ Изменение поведения базового класса в наследниках без сохранения контракта
