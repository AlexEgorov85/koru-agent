# Версионирование промтов

Версионирование промтов в Koru AI Agent Framework обеспечивает контроль изменений, управление жизненным циклом и поддержку нескольких версий промтов одновременно. Эта система позволяет безопасно вносить изменения, откатываться к предыдущим версиям и управлять активными версиями в продакшене.

## Принципы версионирования

Система версионирования основывается на следующих принципах:

### 1. Семантическое версионирование (SemVer)
- Используется формат `MAJOR.MINOR.PATCH` (например, `1.2.3`)
- `MAJOR` - значительные изменения, нарушающие обратную совместимость
- `MINOR` - добавление новой функциональности с сохранением совместимости
- `PATCH` - исправление ошибок без изменения функциональности

### 2. Контроль изменений
- Каждое изменение в логике промта требует новой версии
- Изменения в структуре ответа требуют минимум MINOR-версии
- Критические изменения требуют MAJOR-версии

### 3. Жизненный цикл версии
- Каждая версия проходит статусы: draft → active/shadow → deprecated → archived
- Активные версии могут использоваться в продакшене
- Устаревшие версии продолжают работать, но не рекомендуются к использованию

## Статусы жизненного цикла

### 1. Draft (Черновик)
- Версия находится в разработке
- Не используется в продакшене
- Может изменяться без ограничений
- Пример: `v1.0.0-draft`

### 2. Active (Активная)
- Основная версия, используемая в продакшене
- Полностью протестирована и одобрена
- Может быть только одна активная версия на капабилити/роль
- Пример: `v1.0.0` (без суффикса)

### 3. Shadow (Теневая)
- Экспериментальная версия для A/B тестирования
- Используется для сравнения с активной версией
- Может быть несколько теневых версий
- Пример: `v1.1.0-shadow`

### 4. Deprecated (Устаревшая)
- Версия больше не рекомендуется к использованию
- Продолжает работать для обратной совместимости
- Используется для плавного перехода к новым версиям
- Пример: `v0.9.0-deprecated`

### 5. Archived (Архивная)
- Версия полностью отключена
- Сохранена для исторических целей
- Не может быть использована
- Пример: `v0.8.0-archived`

## Структура версий в файловой системе

Версии промтов организованы в файловой системе следующим образом:

```
prompts/
├── {domain}/
│   └── {capability}/
│       ├── {role}/
│       │   ├── v1.0.0.md
│       │   ├── v1.1.0.md
│       │   ├── v2.0.0-alpha.md      # Пре-релизная версия
│       │   └── v1.0.0-deprecated.md # Устаревшая версия
│       └── _index.yaml
```

Имена файлов соответствуют формату: `v{semver}[suffix].md`

## Управление версиями

### 1. Создание новой версии

При создании новой версии:

1. Определите тип изменений (MAJOR, MINOR, PATCH)
2. Присвойте подходящий номер версии
3. Установите статус `draft`
4. Добавьте описание изменений
5. Протестируйте новую версию
6. Обновите статус до `active` при готовности

### 2. Пример процесса обновления версии

```yaml
# prompts/code_analysis/security_analysis/system/v2.0.0.md
---
provider: openai
role: system
status: draft  # Начинаем с черновика
version: 2.0.0
previous_version: 1.1.5
activation_date: null
deprecation_date: null
author: developer_name
reviewed_by: reviewer_name
change_type: MAJOR
changes: |
  - Полностью переписана логика анализа уязвимостей
  - Изменена структура ожидаемого ответа
  - Добавлена поддержка новых типов уязвимостей
  - Улучшена точность обнаружения
variables:
  - name: code
    type: string
    required: true
    description: "Код для анализа"
expected_response:
  type: object
  properties:
    vulnerabilities:
      type: array
      items:
        type: object
        properties:
          type:
            type: string
          severity:
            type: string
          location:
            type: object
          description:
            type: string
```

### 3. Обновление статуса версии

После тестирования обновляем статус:

```yaml
# prompts/code_analysis/security_analysis/system/v2.0.0.md
---
# ... остальные поля ...
status: active  # Теперь активная версия
activation_date: "2024-01-15T10:00:00Z"  # Дата активации
```

### 4. Устаревание старой версии

При активации новой версии устаревает предыдущая:

```yaml
# prompts/code_analysis/security_analysis/system/v1.1.5.md
---
# ... остальные поля ...
status: deprecated  # Старая версия теперь deprecated
deprecation_date: "2024-01-15T10:00:00Z"  # Дата устаревания
replacement_version: "2.0.0"  # Версия, которая заменяет эту
```

## Стратегии версионирования

### 1. Миграция с одной версии на другую

При миграции рекомендуется:

1. Запустить A/B тестирование с теневой версией
2. Собрать метрики и сравнить результаты
3. Постепенно переводить пользователей на новую версию
4. Устаревать старую версию через установленное время
5. Архивировать старую версию после полного перехода

### 2. Обратная совместимость

Для обеспечения обратной совместимости:

- PATCH-версии не должны менять структуру ожидаемого ответа
- MINOR-версии могут добавлять новые поля в ответ, но не удалять существующие
- MAJOR-версии могут кардинально изменить структуру

### 3. Пре-релизные версии

Для экспериментальных версий используются суффиксы:

- `-alpha` - ранние экспериментальные версии
- `-beta` - версии для бета-тестирования
- `-rc` - релиз-кандидаты

Пример: `v2.0.0-alpha.1`, `v1.5.0-beta`, `v3.0.0-rc.1`

## Инструменты управления версиями

### 1. Загрузчик промтов

Загрузчик автоматически определяет актуальные версии:

```python
from application.services.prompt_loader import PromptLoader

loader = PromptLoader(base_path="./prompts")

# Загрузка всех версий
all_prompts, errors = loader.load_all_prompts()

# Загрузка только активных версий
active_prompts = [p for p in all_prompts if p.status == "active"]

# Загрузка конкретной версии
specific_prompt = next((p for p in all_prompts if p.semantic_version == "2.0.0"), None)
```

### 2. Валидация версий

Система проверяет корректность версионирования:

- Проверка формата номера версии
- Проверка корректности переходов статусов
- Проверка уникальности активных версий
- Проверка ссылок на предыдущие версии

## Лучшие практики

### 1. Планирование изменений

- Планируйте версии заранее
- Документируйте изменения
- Тестируйте совместимость
- Сообщайте пользователям о предстоящих изменениях

### 2. Мониторинг использования

- Отслеживайте использование разных версий
- Анализируйте метрики качества
- Выявляйте проблемы с новыми версиями
- Оценивайте эффективность изменений

### 3. Управление сроками

- Устанавливайте сроки устаревания для старых версий
- Планируйте миграции заранее
- Обеспечьте достаточное время для перехода
- Архивируйте версии после установленного срока

## Пример политики версионирования

```
Политика версионирования для промтов домена code_analysis:

1. PATCH-обновления (безопасные исправления):
   - Исправление опечаток
   - Уточнение инструкций
   - Исправление логических ошибок
   
2. MINOR-обновления (новые возможности):
   - Добавление новых типов анализа
   - Расширение поддерживаемых языков
   - Улучшение существующих возможностей
   
3. MAJOR-обновления (изменения с нарушением совместимости):
   - Изменение структуры ответа
   - Замена алгоритмов анализа
   - Удаление устаревших возможностей

Сроки:
- Активные версии поддерживаются минимум 6 месяцев
- Устаревшие версии поддерживаются минимум 3 месяца
- Архивные версии хранятся минимум 1 год
```

## Интеграция с другими системами

Система версионирования интегрирована с:

- **Системой кэширования**: Кэши учитывают версии промтов
- **Системой логирования**: Логи содержат информацию о версиях
- **Системой мониторинга**: Метрики собираются по версиям
- **Системой A/B тестирования**: Позволяет сравнивать разные версии
- **Системой безопасности**: Контролирует доступ к разным версиям
