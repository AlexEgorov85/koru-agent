## 🔴 Критические риски

### R-001: Производительность ниже требований

**Описание:** Время поиска превышает 1000ms (p95).

**Вероятность:** Средняя (3/5)  
**Влияние:** Высокое (5/5)  
**Уровень риска:** 🔴 Высокий (15/25)

**Причины:**
- Неправильный выбор индекса FAISS
- Большой размер индекса (> 100K векторов)
- Неоптимальная фильтрация

**Митигация:**
- [ ] Использовать IVF индекс с оптимальным nlist
- [ ] Реализовать кэширование частых запросов
- [ ] Оптимизировать пост-фильтрацию
- [ ] Performance тесты на раннем этапе

**План действий:**
1. Начать с Flat индекса для небольших данных
2. Протестировать IVF на 10K, 100K, 1M векторов
3. Добавить метрики производительности
4. Настроить nlist/nprobe эмпирически

---

### R-002: Низкая точность поиска

**Описание:** Recall@10 < 0.90.

**Вероятность:** Средняя (3/5)  
**Влияние:** Высокое (5/5)  
**Уровень риска:** 🔴 Высокий (15/25)

**Причины:**
- Неподходящая модель эмбеддингов
- Неправильный chunking
- Потеря контекста при разбиении

**Митигация:**
- [ ] Выбрать проверенную модель (all-MiniLM-L6-v2)
- [ ] Реализовать chunking с перекрытием
- [ ] Добавить benchmark сценарии
- [ ] Тестировать на реальных данных

**План действий:**
1. Создать тестовый набор документов (100+)
2. Разметить релевантные пары query-document
3. Измерить recall@k для разных моделей
4. Оптимизировать chunk_size/chunk_overlap

---

### R-003: Потеря данных при сбое

**Описание:** Индекс или метаданные повреждаются при сбое.

**Вероятность:** Низкая (2/5)  
**Влияние:** Высокое (5/5)  
**Уровень риска:** 🟡 Средний (10/25)

**Причины:**
- Отсутствие транзакционности
- Повреждение файлов при записи
- Синхронизация индекса и метаданных

**Митигация:**
- [ ] Атомарная запись (temp file + rename)
- [ ] Регулярные бэкапы
- [ ] Валидация при загрузке
- [ ] Event sourcing для восстановления

**План действий:**
1. Реализовать атомарную запись JSON
2. Сохранять индекс с контрольной суммой
3. Добавить команду восстановления
4. Логировать все изменения

## 🟡 Средние риски

### R-004: Сложность интеграции с текущей архитектурой

**Описание:** Интеграция занимает больше времени, чем запланировано.

**Вероятность:** Низкая (2/5)  
**Влияние:** Среднее (3/5)  
**Уровень риска:** 🟡 Низкий (6/25)

**Причины:**
- Несовместимость интерфейсов
- Отсутствие документации
- Зависимости от других компонентов

**Митигация:**
- [ ] Следовать текущей архитектуре (Infrastructure/Application)
- [ ] TDD (тесты перед реализацией)
- [ ] Поэтапная интеграция
- [ ] Mock провайдеры для тестов

**План действий:**
1. Изучить существующие провайдеры (LLM, DB)
2. Создать интерфейсы по аналогии
3. Написать integration тесты
4. Интегрировать через Dependency Injection

- [ ] Регулярные sync с заказчиком

### R-006: Недостаток времени на тесты

**Описание:** Тесты пишутся после реализации или пропускаются.

**Вероятность:** Средняя (3/5)  
**Влияние:** Высокое (5/5)  
**Уровень риска:** 🔴 Высокий (15/25)

**Причины:**
- Давление сроков
- Сложность тестирования
- Отсутствие mock провайдеров

**Митигация:**
- [ ] TDD обязательно (тесты перед кодом)
- [ ] Mock провайдер с первого этапа
- [ ] CI/CD с обязательными тестами
- [ ] Минимальное покрытие 85%

**План действий:**
1. Создать mock провайдер до реализации
2. Написать unit тесты моделей
3. Написать integration тесты интерфейсов
4. Написать e2e тесты сценариев
