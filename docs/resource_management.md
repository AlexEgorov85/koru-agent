# Система управления ресурсами

## Обзор

Система управления ресурсами обеспечивает предзагрузку, кэширование и доступ к ресурсам (промтам и контрактам) для всех компонентов системы (навыков, инструментов, сервисов).

## Архитектурные принципы

### 1. Предзагрузка → Кэш → Выполнение
- Все ресурсы предзагружаются при инициализации компонента
- Ресурсы хранятся в изолированном кэше каждого компонента
- Во время выполнения компонент НЕ обращается к внешним сервисам за ресурсами

### 2. Изолированные кэши
- Каждый компонент имеет собственный изолированный кэш
- Кэши не зависят друг от друга
- Это обеспечивает изоляцию и безопасность

### 3. Единая точка инициализации
- Все ресурсы загружаются через `ComponentConfig`
- Инициализация происходит в `BaseComponent.initialize()`
- После инициализации компонент полностью автономен

## Структура ресурсов

### Промты
- Хранятся в `_cached_prompts: Dict[str, str]`
- Ключ: имя capability
- Значение: текст промта

### Входные контракты
- Хранятся в `_cached_input_contracts: Dict[str, Dict]`
- Ключ: имя capability
- Значение: схема входного контракта

### Выходные контракты
- Хранятся в `_cached_output_contracts: Dict[str, Dict]`
- Ключ: имя capability
- Значение: схема выходного контракта

## Процесс загрузки ресурсов

### 1. Конфигурация
Ресурсы определяются в конфигурации компонента:

```yaml
app_config:
  prompt_versions:
    "planning.create_plan": "v1.0.0"
    "book_library.search_books": "v1.0.0"
  input_contract_versions:
    "planning.create_plan": "v1.0.0"
    "book_library.search_books": "v1.0.0"
  output_contract_versions:
    "planning.create_plan": "v1.0.0"
    "book_library.search_books": "v1.0.0"
```

### 2. Предзагрузка
- При инициализации `ApplicationContext` все ресурсы загружаются из хранилища
- Ресурсы группируются по типам и версиям
- Дубликаты устраняются для оптимизации

### 3. Обогащение конфигурации
- Загруженные ресурсы добавляются в `ComponentConfig`
- Каждый компонент получает только свои ресурсы
- Недостающие ресурсы заменяются заглушками

### 4. Инициализация компонентов
- Компоненты загружают свои ресурсы из `ComponentConfig`
- Ресурсы кэшируются в изолированных кэшах компонентов
- Компонент готов к работе без внешних зависимостей

## Использование в компонентах

### Доступ к промтам
```python
# Внутри компонента
prompt_text = self.get_prompt("planning.create_plan")
```

### Доступ к контрактам
```python
# Входной контракт
input_schema = self.get_input_contract("planning.create_plan")

# Выходной контракт
output_schema = self.get_output_contract("planning.create_plan")
```

### Безопасный доступ
```python
# Безопасное получение с проверкой срока действия
prompt_text = self.get_cached_prompt_safe("planning.create_plan")
input_schema = self.get_cached_input_contract_safe("planning.create_plan")
output_schema = self.get_cached_output_contract_safe("planning.create_plan")
```

## Управление кэшем

### Инвалидация кэша
```python
# Инвалидация всех кэшей
self.invalidate_cache()

# Инвалидация конкретного типа кэша
self.invalidate_cache('prompts')

# Инвалидация конкретного элемента
self.invalidate_cache('prompts', 'planning.create_plan')
```

### Настройка TTL
```python
# Установка времени жизни кэша (в секундах)
self._cache_ttl_seconds = 3600  # 1 час
```

## Диагностика и мониторинг

### Health Check
```python
# Проверка состояния всех компонентов
health_report = await context.health_check()
```

Отчет включает:
- Общий статус системы
- Статус каждого компонента
- Количество загруженных ресурсов
- Время выполнения проверки

## Конфигурация

### Регистрация ресурсов
Ресурсы регистрируются в `registry.yaml`:

```yaml
skills:
  planning:
    enabled: true
    prompt_versions:
      planning.create_plan: "v1.0.0"
    input_contract_versions:
      planning.create_plan: "v1.0.0"
    output_contract_versions:
      planning.create_plan: "v1.0.0"
```

### Версионирование
- Каждый ресурс имеет версию
- Версии позволяют управлять изменениями
- Поддерживаются разные версии для разных компонентов

## Best Practices

### 1. Минимизация обращений к внешним сервисам
- Все ресурсы должны быть предзагружены
- Во время выполнения НЕ должно быть обращений к внешним сервисам
- Использовать изолированные кэши

### 2. Обработка отсутствующих ресурсов
- Всегда проверять наличие ресурсов
- Использовать безопасные методы доступа
- Обеспечивать graceful degradation

### 3. Мониторинг состояния
- Регулярно выполнять health check
- Логировать состояние загрузки ресурсов
- Отслеживать производительность

## Troubleshooting

### Отладка загрузки ресурсов
1. Проверьте `registry.yaml` на наличие необходимых определений
2. Убедитесь, что файлы ресурсов существуют в соответствующих директориях
3. Проверьте логи инициализации на наличие ошибок загрузки
4. Используйте health check для диагностики состояния

### Распространенные проблемы
- Отсутствующие файлы ресурсов
- Неправильные версии в конфигурации
- Ошибки формата файлов ресурсов
- Проблемы с правами доступа к файлам
