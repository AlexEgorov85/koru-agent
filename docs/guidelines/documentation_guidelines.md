# Руководство по документированию проекта

## Общие принципы

### Структура документации
- Все документы должны находиться в папке `docs/`
- Использовать подпапки для логической группировки
- Поддерживать иерархию: `docs/категория/подкатегория/документ.md`

### Стиль написания
- Использовать русский язык для всех комментариев и документации
- Писать предложения с заглавной буквы и заканчивать точкой
- Избегать избыточных комментариев для тривиального кода
- Для сложной логики добавлять пояснения

## Типы документов

### Описание компонентов
- Назначение компонента
- Архитектурная роль
- Зависимости
- Примеры использования
- Статус (в разработке, стабильна, требует улучшения)

### Архитектурные решения
- Использовать шаблон из `docs/templates/architecture_decision_template.md`
- Добавлять в папку `docs/architecture/decisions/`
- Нумеровать последовательно (ADR-001, ADR-002 и т.д.)

### Руководства
- Пошаговые инструкции
- Примеры кода
- Скриншоты (при необходимости)
- Часто задаваемые вопросы

## Форматирование

### Заголовки
- Использовать `#`, `##`, `###` для иерархии
- Заголовок документа - `#`
- Разделы - `##`
- Подразделы - `###`

### Код
- Использовать синтаксис highlighting для кода
- Указывать язык после открывающих кавычек: ```python, ```bash и т.д.
- Пример:
  ```python
  def example_function():
      """Пример функции с документацией"""
      pass
  ```

### Ссылки
- Использовать относительные пути для внутренних ссылок
- Проверять актуальность всех ссылок
- Указывать пояснения к ссылкам

## Процесс обновления документации

### При добавлении новой функциональности
1. Создать документ в соответствующей подпапке
2. Добавить ссылку в `docs/SUMMARY.md`
3. Указать статус в `docs/components_status.md`
4. Обновить `README.md` при необходимости

### При изменении существующей функциональности
1. Обновить соответствующий документ
2. Проверить все примеры кода
3. Обновить статус в `docs/components_status.md`

### При удалении функциональности
1. Отметить как устаревшую в документе
2. Обновить статус в `docs/components_status.md`
3. Удалить или обновить ссылки в других документах

## Проверка качества документации

### Перед коммитом
- Проверить орфографию
- Убедиться, что все ссылки работают
- Проверить примеры кода на актуальность
- Убедиться, что документация соответствует реализации

### Регулярная проверка
- Ежемесячно проверять актуальность документов
- Обновлять статусы компонентов
- Удалять устаревшие документы
- Обновлять `docs/SUMMARY.md`

## Шаблоны

### Для новых компонентов
Использовать шаблон из `docs/templates/component_template.md`

### Для архитектурных решений
Использовать шаблон из `docs/templates/architecture_decision_template.md`

### Для руководств
- Введение
- Предпосылки
- Пошаговая инструкция
- Заключение
- Ссылки на дополнительные ресурсы
