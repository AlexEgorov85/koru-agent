# Архитектура кэширования промптов и контрактов

## Обзор

Архитектура кэширования промптов и контрактов обеспечивает однократную загрузку ресурсов при инициализации компонентов и последующее использование кэша без обращений к хранилищу во время выполнения.

## Компоненты архитектуры

### 1. AgentConfig
- Модель конфигурации агента
- Поддерживает автоматическое разрешение из активных версий
- Фиксирует версии промптов и контрактов для воспроизводимости

### 2. BaseSkill с кэшированием
- Расширен базовый класс навыков
- Поддержка однократной загрузки промптов при инициализации
- Метод `get_prompt()` для получения из кэша

### 3. AgentStrategyInterface с кэшированием
- Расширена базовая стратегия
- Поддержка однократной загрузки промптов при инициализации
- Метод `get_prompt()` для получения из кэша

### 4. BaseTool
- Базовый класс для инструментов с кэшированием
- Поддержка однократной загрузки промптов и контрактов через наследование от BaseComponent
- Методы `get_prompt()` и `get_contract()` для получения из кэша

### 5. CachedBaseService
- Базовый класс для сервисов с кэшированием
- Поддержка однократной загрузки промптов и контрактов
- Методы `get_prompt()` и `get_contract()` для получения из кэша

### 6. AgentFactory
- Обновлена фабрика агентов
- Инициализирует все компоненты с кэшированием
- Гарантирует однократную загрузку ресурсов

## Гарантии архитектуры

| Гарантия | Реализация |
|----------|------------|
| **Однократная загрузка** | Все промпты/контракты загружаются один раз при инициализации компонента |
| **Отсутствие обращений к хранилищу во время выполнения** | Все компоненты используют кэш после инициализации |
| **Воспроизводимость** | Конфигурация фиксирует версии ресурсов |
| **Безопасность** | Защита от использования компонентов до инициализации |
| **Разделение ответственности** | Системные ресурсы vs компоненты агента |

## Использование

Каждый компонент должен:
1. Наследоваться от соответствующего базового класса с кэшированием
2. Реализовать методы для указания требуемых промптов/контрактов
3. Вызывать `initialize_with_config()` при инициализации
4. Использовать `get_prompt()` и `get_contract()` для получения ресурсов

## Преимущества

- **Производительность**: Отсутствие обращений к хранилищу во время выполнения
- **Надежность**: Защита от проблем с доступом к хранилищу во время выполнения
- **Воспроизводимость**: Фиксированные версии ресурсов для каждой сессии
- **Масштабируемость**: Уменьшение нагрузки на хранилище промптов/контрактов
