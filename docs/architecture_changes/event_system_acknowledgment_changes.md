# Архитектурные изменения: Механизм подтверждения доставки событий

## Обзор изменений

Добавлен механизм подтверждения доставки событий в шину событий (EventSystem) для обеспечения надежной доставки важных сообщений с автоматическими повторными попытками.

## Архитектурные компоненты

### 1. Интерфейс IEventPublisher (доменный слой)

**Файл**: `domain/abstractions/event_types.py`

**Изменения**:
- Добавлен метод `publish_with_ack()` - публикация события с отслеживанием подтверждения
- Добавлен метод `subscribe_with_ack()` - подписка с подтверждением обработки
- Реализация по умолчанию в абстрактном классе для обеспечения обратной совместимости

### 2. Базовая реализация EventSystem (инфраструктурный слой)

**Файл**: `infrastructure/event_system.py`

**Изменения**:
- Добавлены реализации методов `publish_with_ack()` и `subscribe_with_ack()` 
- Обеспечена обратная совместимость с существующим кодом
- Минимальная логика подтверждения (базовая реализация)

### 3. Расширенная реализация AcknowledgedEventSystem (инфраструктурный слой)

**Файл**: `infrastructure/event_system_with_ack.py`

**Новый компонент**:
- Полноценная реализация механизма подтверждения доставки
- Отслеживание статуса событий (PENDING, CONFIRMED, FAILED, RETRYING)
- Автоматические повторные попытки для неподтвержденных событий
- Фоновый процесс мониторинга retry
- Настройка параметров retry (max_retries, retry_delay)

### 4. Документация

**Новый компонент**:

- Архитектурные преимущества

### 5. Тесты

**Новый компонент**:
- Тесты для новой функциональности

### 6. Примеры

## Архитектурные принципы

### 1. Обратная совместимость
- Существующий код продолжает работать без изменений
- Новые методы добавлены как опциональные расширения
- Базовая реализация в EventSystem обеспечивает совместимость

### 2. Инверсия зависимостей
- Интерфейс IEventPublisher определен в доменном слое
- Расширенная реализация остается в инфраструктурном слое
- Прикладной код может зависеть от абстракции, а не от конкретной реализации

### 3. Открытость/закрытость
- Система открыта для расширения (новые методы подтверждения)
- Закрыта для модификации (существующий интерфейс не изменен)

### 4. Минимальная сложность
- Новая функциональность опциональна
- Простая интеграция с существующим кодом
- Четкое разделение ответственностей

## Влияние на архитектуру

### Положительные изменения:
1. **Надежность**: гарантия обработки важных событий
2. **Мониторинг**: возможность отслеживания статуса событий
3. **Автоматизация**: автоматические повторные попытки
4. **Совместимость**: без изменений в существующем коде

### Потенциальные риски:
1. **Производительность**: дополнительная нагрузка на систему от отслеживания
2. **Сложность**: увеличение количества компонентов
3. **Ресурсы**: фоновый процесс мониторинга retry

## Рекомендации по использованию

### Когда использовать:
- Для критически важных событий
- Когда необходимо гарантировать обработку
- При работе с внешними системами
- Для асинхронных операций с долгим временем выполнения

### Когда не использовать:
- Для вспомогательных/информационных событий
- Когда потеря отдельных событий не критична
- В высоконагруженных сценариях без необходимости подтверждения

## Миграция

Существующий код не требует изменений:

```python
# Старый код продолжает работать
event_system = get_event_system()
await event_system.publish(EventType.INFO, "source", data)
```

Для использования подтверждения:

```python
# Новый код с подтверждением
event_system = get_ack_event_system()
event_id = await event_system.publish_with_ack(EventType.CRITICAL, "source", data)
```

## Заключение

Механизм подтверждения доставки событий успешно интегрирован в архитектуру системы с соблюдением принципов чистой архитектуры и обеспечением обратной совместимости. Реализация предоставляет надежную доставку событий с минимальным влиянием на существующую инфраструктуру.
