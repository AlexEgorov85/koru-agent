Система построена на принципах чёткого разделения ответственности, изоляции агентов и декларативной конфигурации.### Назначение- **Автономные агенты**: Выполнение сложных задач через цикл ReAct (Reason + Act)- **Планирование**: Декомпозиция целей на последовательности действий- **Интеграция**: Поддержка различных LLM-провайдеров и баз данных- **Безопасность**: Изоляция между агентами, валидация контрактов, sandbox-режим### Ключевые возможности- ✅ **Reasoning-циклы**: ReAct-паттерн с валидацией действий- ✅ **Планирование**: Иерархическая декомпозиция задач- ✅ **Изоляция**: Каждый агент имеет собственный контекст и кэши- ✅ **Версионирование**: Промпты, контракты и паттерны поведения версионируются- ✅ **Структурированный вывод**: Нативная валидация провайдеров + резервная системная- ✅ **Предзагрузка ресурсов**: 0 обращений к хранилищу во время выполнения---## 🏗️ Архитектурные принципы### 1. Разделение слоёв```Infrastructure Layer (общий для всех агентов)    ↓Application Layer (изолирован на агента)    ↓Session Layer (данные сессии)```### 2. Декларация ≠ Данные ≠ Реализация- **Декларация**: Манифесты компонентов в `data/manifests/`- **Данные**: Промпты и контракты в `data/prompts/`, `data/contracts/`- **Реализация**: Код компонентов в `core/`### 3. Предзагрузка → Кэш → Выполнение- Все ресурсы загружаются при инициализации- Компоненты используют изолированные кэши- Во время выполнения нет обращений к внешним хранилищам### 4. Контрактная валидация- Входные/выходные данные валидируются по схемам- Поддержка версионирования контрактов- Ошибки валидации не приводят к падению агента---## 📐 Слои системы### Инфраструктурный слой (Infrastructure)**Назначение**: Управление тяжёлыми ресурсами, общими для всех агентов```mermaidgraph TD    IC[InfrastructureContext]    IC --> PF[ProviderFactory]    IC --> RR[ResourceRegistry]    IC --> LM[LifecycleManager]    IC --> EB[EventBus]    PF --> LLM[LLM Providers]    PF --> DB[Database Providers]    RR --> PS[PromptStorage]    RR --> CS[ContractStorage]```**Компоненты**:- `InfrastructureContext` — общий контекст инфраструктуры- `ProviderFactory` — фабрика провайдеров (LLM, БД)- `ResourceRegistry` — реестр ресурсов- `LifecycleManager` — управление жизненным циклом- `EventBus` — шина событий системы### Прикладной слой (Application)**Назначение**: Изолированная логика конкретного агента```mermaidgraph TD    AC[ApplicationContext]    AC --> CR[ComponentRegistry]    AC --> CC[ComponentConfig]    AC --> IC[InfrastructureContext link]    CR --> S[Services]    CR --> SK[Skills]    CR --> T[Tools]```**Компоненты**:- `ApplicationContext` — изолированный контекст агента- `ComponentRegistry` — реестр компонентов агента- `ComponentConfig` — конфигурация компонента- `BaseService` — базовый класс сервиса- `BaseSkill` — базовый класс навыка- `BaseTool` — базовый класс инструмента### Слой сессии (Session)**Назначение**: Хранение данных сессии агента```mermaidgraph TD    SC[SessionContext]    SC --> DC[DataContext]    SC --> StC[StepContext]    DC --> CI[ContextItem]    StC --> AS[AgentStep]```**Компоненты**:- `SessionContext` — контекст сессии- `DataContext` — данные сессии (истина)- `StepContext` — шаги reasoning-цикла---## 🔗 Контексты### InfrastructureContextОбщий для всех агентов контекст инфраструктуры.```pythonclass InfrastructureContext:    provider_factory: ProviderFactory    resource_registry: ResourceRegistry    lifecycle_manager: LifecycleManager    event_bus: EventBus    config: SystemConfig```**Особенности**:- Создаётся один на приложение- Содержит тяжёлые ресурсы (модели LLM, подключения к БД)- Доступен для чтения из ApplicationContext### ApplicationContextИзолированный контекст каждого агента.```pythonclass ApplicationContext:    component_registry: ComponentRegistry    infrastructure_context: InfrastructureContext  # read-only    config: AppConfig```**Особенности**:- Создаётся для каждого агента отдельно- Содержит изолированные кэши промптов и контрактов- Обеспечивает изоляцию между агентами### SessionContextКонтекст сессии агента.```pythonclass SessionContext:    data_context: DataContext      # вся истина, сырые данные    step_context: StepContext      # reasoning / шаги    current_plan_item_id: str      # ссылка на текущий элемент плана```**Особенности**:- Append-only для данных- Хранит историю взаимодействий- Связывает шаги с элементами плана---## 🧩 Компоненты### Типы
