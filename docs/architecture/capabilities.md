# Архитектура «Декларация ≠ Реализация ≠ Данные»

## Принципы архитектуры

```
Capability = ДЕКЛАРАЦИЯ (ЧТО можно сделать) → только метаданные
Contract   = ДАННЫЕ (какие параметры нужны) → версионируемые ресурсы
Skill      = РЕАЛИЗАЦИЯ (КАК это сделать) → бизнес-логика без прямого доступа к хранилищу
```

## Структура компонентов

### 1. Capability (Декларация)

Класс `Capability` теперь содержит только метаданные о том, **что** можно сделать:

```python
class Capability(BaseModel):
    name: str                    # Уникальное имя capability
    description: str             # Человекочитаемое описание
    skill_name: str              # Имя навыка, которому принадлежит capability
    visiable: bool               # Видимость возможности
    supported_strategies: List[str]  # Поддерживаемые стратегии
    meta: Dict[str, Any]         # Метаданные для отладки и версионирования
```

**Важно**: Поле `parameters_schema` больше не используется в `Capability`. 
Все схемы параметров теперь находятся в **Contract**.

### 2. Contract (Данные)

Контракты содержат схемы данных и управляются через `ContractService`:

- **Input Contract**: Определяет схему входных параметров для capability
- **Output Contract**: Определяет схему выходных данных для capability
- **Версионирование**: Каждый контракт имеет версию (например, `v1.0.0`)
- **Направление**: Контракты различаются по направлению (`input`/`output`)

### 3. Skill (Реализация)

Навыки содержат логику выполнения и работают только с предзагруженными ресурсами:

- **Предзагрузка**: Все промпты и контракты загружаются при инициализации
- **Кэширование**: Навыки используют изолированные кэши для доступа к ресурсам
- **Без обращений к ФС**: После инициализации навыки не обращаются к файловой системе
- **ComponentConfig**: Навыки получают `ComponentConfig` с конкретными версиями промптов и контрактов

## Жизненный цикл инициализации

### Этапы инициализации:

1. **Создание InfrastructureContext** (один на всё приложение)
2. **Создание ApplicationContext** (на агента/сессию)
3. **Инициализация инфраструктурных сервисов** (ContractService, PromptService)
4. **ПРЕДЗАГРУЗКА всех контрактов** через `contract_service.preload_contracts(agent_config)`
5. **ПРЕДЗАГРУЗКА всех промптов** через `prompt_service.preload_prompts(agent_config)`
6. **Инициализация навыков** (уже с заполненными кэшами)
7. **Проверка готовности** через `registry.verify_all_resources_preloaded()`

### Проверка готовности:

```python
# В ApplicationContext
if not application_context.is_fully_initialized():
    raise InitializationError(
        "Система не готова: не все ресурсы предзагружены"
    )

# В AgentRuntime
await self._verify_readiness()
```

## Безопасный доступ к ресурсам

### В навыках запрещено:

```python
# ЗАПРЕЩЕНО:
schema = self.contract_service.get_contract_schema(...)  # ← ОШИБКА!
prompt = self.prompt_service.get_prompt(...)             # ← ОШИБКА!
```

### В навыках разрешено:

```python
# РАЗРЕШЕНО:
schema = self.get_input_contract(capability.name)        # ← из кэша
schema = self.get_output_contract(capability.name)       # ← из кэша
prompt = self.get_prompt(capability.name)                # ← из кэша
```

## Запрещённые паттерны

1. **Обращения к файловой системе во время выполнения**:
   - Все ресурсы должны быть предзагружены при инициализации
   - Навыки не должны обращаться к файловой системе во время выполнения

2. **Использование устаревших полей**:
   - Не использовать `Capability.parameters_schema`
   - Не использовать прямой доступ к сервисам из методов выполнения

3. **Изменение контрактов во время выполнения**:
   - Контракты фиксируются при инициализации
   - Невозможно изменить контракты во время выполнения сессии
