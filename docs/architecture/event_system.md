# Система событий

## Архитектура

Система событий была упрощена в соответствии с принципами чистой архитектуры:

- **Один интерфейс**: `IEventPublisher` в доменном слое
- **Одна основная реализация**: `EventSystem` в инфраструктурном слое
- **Композиция вместо наследования**: фильтры, валидаторы и рейт-лимитеры подключаются через композицию
- **Централизованная публикация**: события публикуются только из `AgentRuntime` и `SystemOrchestrator`

## Компоненты

### IEventPublisher (Интерфейс)
- Расположение: `domain/abstractions/event_types.py`
- Ответственность: определение контракта для публикации событий

### EventSystem (Реализация)
- Расположение: `infrastructure/event_system.py`
- Ответственность: управление событиями, подписками, обработка

### Компоненты фильтрации
- Расположение: `infrastructure/event_filters/`
- Компоненты: `SecurityEventFilter`, `SizeLimitFilter`, `RateLimiter`

## Использование

### В AgentRuntime (центральная точка публикации)
```python
# События теперь публикуются централизованно
await self.event_publisher.publish(EventType.INFO, "AgentRuntime", {"message": "Task completed"})
```

### В SystemOrchestrator (центральная точка публикации)
```python
# События также могут публиковаться из оркестратора
await self.event_system.publish(EventType.SYSTEM, "SystemOrchestrator", {"status": "initialized"})
```

## Миграция

Старые компоненты помечены как устаревшие:
- `IEventPublisherFactory` → используйте `EventSystem` напрямую
- `EventPublisherAdapter` → используйте `EventSystem` напрямую
- `EventBusAdapter` → используйте `EventSystem` напрямую

## Принципы

1. **Минимальная абстракция**: Один интерфейс + одна реализация
2. **Единая точка ответственности**: Публикация событий только из двух мест
3. **Композиция вместо наследования**: Фильтры и валидаторы подключаются через композицию
4. **Четкое разделение обязанностей**: Каждый компонент имеет одну роль
