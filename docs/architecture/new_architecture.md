# Документация по новой архитектуре агента

## Обзор изменений

В рамках улучшения архитектуры были реализованы следующие ключевые изменения:

### 1. Профили работы (prod/sandbox)

Добавлена поддержка двух профилей работы:
- `prod` - продакшн режим с жесткой валидацией версий
- `sandbox` - режим разработки с гибкой валидацией

#### Валидация версий
- В `prod` режиме разрешены только версии со статусом `active`
- В `sandbox` режиме разрешены версии со статусами `draft` и `active`, `archived` - с предупреждением

### 2. Изоляция кэшей

Каждый `ApplicationContext` теперь имеет:
- Изолированные кэши промптов и контрактов
- Изолированные экземпляры сервисов
- Защиту от раннего доступа к неинициализированным ресурсам

### 3. Управление версиями

#### Оверрайды версий
- Метод `set_prompt_override()` позволяет переопределять версии промптов
- Оверрайды разрешены только в `sandbox` режиме
- Для `prod` режима выбрасывается исключение

#### Горячая замена версий
- Метод `clone_with_version_override()` создает новый контекст с обновленными версиями
- Не требует пересоздания инфраструктурных компонентов
- Обеспечивает быстрое переключение версий

### 4. Структура данных

Все данные (промпты и контракты) теперь находятся в единой директории:
```
data/
├── prompts/
│   ├── application/
│   │   ├── planning/
│   │   └── book_library/
│   └── tools/
│       └── sql_tool/
└── contracts/
    ├── application/
    └── tools/
```

### 5. Конфигурация

#### Обратная совместимость
- Поле `type_provider` автоматически преобразуется в `provider_type`
- Выводится предупреждение при использовании устаревшего поля
- Поддержка старых конфигурационных файлов

### 6. ComponentConfig vs AppConfig

#### Разделение ответственности
- `AppConfig` - глобальная конфигурация приложения, содержит общие параметры для всей системы
- `ComponentConfig` - локальная конфигурация компонента, содержит специфичные версии промптов и контрактов

#### Использование
- `AppConfig` загружается из `data/registry.yaml` и используется для настройки системы в целом
- `ComponentConfig` создается для каждого компонента отдельно и обеспечивает изоляцию между агентами
- Компоненты (навыки, инструменты, сервисы) получают `ComponentConfig` для предзагрузки своих ресурсов

## Примеры использования

### Создание контекста с профилем

```python
from core.application.context.application_context import ApplicationContext

# Продакшн контекст
prod_context = ApplicationContext(
    infrastructure_context=infra,
    config=agent_config,
    profile="prod"  # Только активные версии
)

# Песочница
sandbox_context = ApplicationContext(
    infrastructure_context=infra,
    config=agent_config,
    profile="sandbox"  # Разрешены draft версии
)
```

### Управление версиями в песочнице

```python
# Установка оверрайда версии (только для песочницы)
sandbox_context.set_prompt_override("planning", "v1.1.0")

# Проверка, что оверрайд работает
prompt = sandbox_context.get_prompt("planning")  # Вернет v1.1.0
```

### Горячая замена версий

```python
# Создание нового контекста с обновленными версиями
new_context = await old_context.clone_with_version_override(
    prompt_overrides={"planning": "v2.0.0"},
    contract_overrides={"planning": "v2.0.0"}
)
```

## Архитектурные гарантии

1. **Изоляция агентов** - каждый контекст имеет свои кэши
2. **Безопасность продакшна** - строгая валидация версий
3. **Гибкость разработки** - поддержка черновиков в песочнице
4. **Быстрое переключение** - клонирование контекстов без перезагрузки инфраструктуры
5. **Обратная совместимость** - поддержка старых конфигурационных файлов
