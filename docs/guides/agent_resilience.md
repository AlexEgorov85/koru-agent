# Устойчивость агентного цикла: практические правила

## Зачем нужен этот документ

В проекте добавлены механизмы, которые защищают агента от:
- зацикливания на одном и том же действии,
- накопления пустых результатов,
- неэффективных SQL-диагностик,
- деградации качества рассуждения в длинных сессиях.

Документ фиксирует архитектурные принципы и ожидаемое поведение.

---

## 1. AgentState: total vs consecutive метрики

`AgentState` хранит **два класса метрик**:

1. **Total** метрики — для аналитики (`total_empty_results`).
2. **Consecutive** метрики — для решений policy/stop
   (`consecutive_empty_results`, `consecutive_repeated_actions`).

Почему это важно:
- случайный единичный пустой ответ не должен останавливать сессию,
- останавливать нужно именно «полосу» неудач.

---

## 2. Повторы считаем по signature, а не только по action

Повтор определяется по `action_signature = action + normalized_parameters`.

Это означает:
- `sql_tool.execute_query(query="...2024")` и
  `sql_tool.execute_query(query="...2025")`
  считаются **разными** шагами,
- policy не блокирует валидные повторные вызовы с новыми параметрами.

---

## 3. Policy блокирует только устойчивые патологии

Текущее правило блокировки:
- `repeat_action` — если уже был такой же `action_signature` в недавнем окне,
- `empty_loop` — если `consecutive_empty_results >= 2`.

Рекомендуется держать эти пороги консервативными и изменять только после анализа логов.

---

## 4. SQL empty recovery: универсальный, типо-ориентированный

При пустом SQL-результате диагностика строится не только для даты/года.

Подход:
1. Эвристически разбирается `WHERE`.
2. Определяется тип первого фильтра (`string/number/date/unknown`).
3. Формируется эффективная подсказка:
   - `string`: `LOWER(TRIM(...))` + `LIKE` для поиска вариантов значений,
   - `number`: `MIN/MAX` + распределение значений,
   - `date`: диапазон + распределение по месяцам,
   - fallback для неизвестных случаев.

---

## 5. Adaptive hints в prompt

PromptBuilder добавляет блок `АДАПТИВНЫЕ ПОДСКАЗКИ` на основе state/observation:
- ошибки,
- пустые результаты,
- повторы,
- SQL-specific issues.

Важно:
- подсказки должны быть краткими и конкретными,
- не дублировать контракт,
- не заменять системные policy-ограничения.

---

## 6. Что проверять в тестах

Минимальный набор:
1. SQL empty + year filter -> `sql_filter_mismatch` + `sql_year_filter_mismatch`.
2. SQL empty + string filter -> `sql_filter_mismatch` + hint с `LOWER(TRIM)`/`LIKE`.
3. Non-SQL empty -> дефолтный hint.
4. Повтор action с разными параметрами -> не блокируется как `repeat_action`.

---

## 7. Ограничения текущей реализации

- SQL-анализ остаётся эвристическим (regex), не полноценным SQL parser.
- Для сложных запросов (`CTE`, вложенные `WHERE`, диалекты) возможны неточности.
- SQL recovery уже вынесен в отдельный анализатор `SQLRecoveryAnalyzer`,
  чтобы не размазывать ответственность внутри `AgentRuntime`.
- При дальнейшем росте сложности рекомендуется заменить regex-эвристики
  на parser-level решение (например, через sqlglot/sqlparse + AST).
